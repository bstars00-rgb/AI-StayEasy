# StayEasy — 실서비스 전환 & Google AdSense 승인 기획서

> 목표: 현재의 "데모 프로토타입"을 **실제 작동하는 콘텐츠 사이트**로 전환하고,
> **Google AdSense 승인**을 통과해 광고 수익(비OTA BM의 핵심)을 실현한다.

---

## 0. 한 줄 결론 (현실 점검)

지금 상태로 AdSense를 신청하면 **거의 확실히 거절**된다. 가장 큰 이유 두 가지:

1. **콘텐츠 진정성 부족** — 호텔이 전부 가짜다("(Sample)", "fictional", "mock data",
   "demo" 라벨이 사이트 곳곳에 노출). AdSense의 1순위 거절 사유가 **"가치 없는 콘텐츠 /
   제작 중(Low value / Under construction)"** 이다.
2. **도메인 미소유** — `bstars00-rgb.github.io/AI-StayEasy/` 는 GitHub 소유 서브경로다.
   AdSense는 **본인이 소유한 최상위 도메인**을 요구한다(무료 호스팅 서브도메인은 사실상 불가).

→ 즉 "광고를 붙이기 전에, **진짜 가치 있는 오리지널 콘텐츠 사이트**가 먼저 되어야 한다."
   다행히 우리 포지셔닝(직접예약 가이드/큐레이션)은 **오리지널 에디토리얼 콘텐츠**와
   궁합이 좋다. 가짜 호텔을 진짜처럼 꾸미는 게 아니라, **진짜 유용한 가이드 콘텐츠**로
   승부하는 방향으로 간다.

## 1. AdSense 승인 핵심 요건 & 우리 대응

| # | AdSense 요건 | 현재 | 대응 |
|---|---|---|---|
| 1 | **본인 소유 도메인** (HTTPS) | ❌ github.io 서브경로 | 커스텀 도메인 구매(예 `stayeasy.asia`) → Vercel/Cloudflare Pages 연결 |
| 2 | **오리지널·고품질 콘텐츠 충분량** | ❌ 가짜 호텔, Sample 라벨 | "Sample/demo" 전면 제거 + **오리지널 가이드 15~25편**(직접예약법, 도시별 가이드, 시즌 팁) |
| 3 | **개인정보처리방침**(쿠키·제3자 광고·Google 명시) | ❌ 없음 | `/privacy` 페이지 신규 — 쿠키/AdSense/DoubleClick/opt-out 링크 의무 포함 |
| 4 | **About / Contact** (실제 연락 수단) | △ About만 | `/contact`(실제 이메일·폼), About 보강 |
| 5 | **이용약관 / 면책** | △ 데모 면책만 | `/terms` 추가, 비OTA·정보제공 성격 명시 |
| 6 | **사이트 완성도** (broken link·제작중 표시 없음) | △ | "준비중/coming soon" 표현 점검, 모든 내비 작동 |
| 7 | **크롤링 가능 + 색인** | ❌ sitemap/robots 없음 | `robots.txt`, `sitemap.xml`, 페이지별 메타/OG(일부 존재) 강화 |
| 8 | **ads.txt** (승인 후) | ❌ | 승인 후 `ads.txt`에 `google.com, pub-XXXX, DIRECT, f08c47fec0942fa0` |
| 9 | **정책 위반 콘텐츠 없음** | ✅(대체로) | Unsplash 사진은 라이선스 OK, picsum 플레이스홀더는 "콘텐츠"로 부적합 → 실사진 확대 |
| 10 | **트래픽·사이트 나이** | ❌ 신규 | 일부 국가(인도 등) 6개월 요건. 신청 전 색인+소량 트래픽 확보 권장 |

> ⚠️ **광고 배치 정책**: 광고를 콘텐츠/내비처럼 보이게 하면 안 됨. 우리 "Sponsored" 라벨,
> 바우처/스폰서 카드와 **AdSense 디스플레이 광고는 시각적으로 명확히 구분**해야 한다.
> (스폰서 콘텐츠 ≠ AdSense 광고. 둘 다 "광고"지만 정책상 표기·구분 기준이 다름.)

## 2. "실제 작동하는 웹사이트" 요건

데모와 실서비스의 차이 = **데이터가 진짜로 저장/공유되고, 폼이 실제로 동작**하는가.

| 영역 | 데모(현재) | 실서비스 목표 |
|---|---|---|
| 데이터 | 정적 + localStorage | 백엔드 DB(이미 Fastify+Prisma 스캐폴드 보유) 실배포 |
| 호텔 등록 | 내 브라우저에만 저장 | `POST /partners` → DB 저장, 운영자 승인 워크플로우 |
| 파트너 문의 / 뉴스레터 | UI만 | 실제 이메일 발송(예: Resend/SES) + DB 리드 저장 |
| 콘텐츠 | 가짜 호텔 | 오리지널 가이드(MDX/CMS) + 검증된 실제/실명 호텔 일부 |
| 호스팅 | GitHub Pages(정적) | Vercel(프론트) + Render/Fly(API) + 관리 도메인 |
| 분석 | 없음 | GA4(또는 Plausible) — AdSense 연동·트래픽 측정 |

## 3. BM 정합성 — 비OTA 광고 모델에서 AdSense의 위치

- StayEasy 매출 = ① **AdSense 디스플레이 광고**(콘텐츠 페이지) + ② **파트너 스폰서십**
  (홈/도시 상단 노출·스폰서 기사·바우처) + ③ 향후 데이터/리포트.
- **예약 수수료는 받지 않는다**(공식 사이트로 보냄). 이 정체성은 유지.
- AdSense는 "정보성 콘텐츠가 풍부한 사이트"에 잘 맞는다 → **가이드/매거진 영역**이
  AdSense의 주 무대, **호텔 디렉터리**는 스폰서십의 주 무대로 역할 분담.

## 4. 단계별 로드맵 (우선순위 + 책임 주체)

표기: 🤖 = 코드로 내가 처리 가능 / 🧑 = 사용자(계정·결제·실데이터) 필요

### Phase 0 — 콘텐츠 진정성 (AdSense 1순위 차단요인 제거) 🤖
- [ ] 모든 "(Sample)/demo/mock/fictional/준비중" 표현 제거 또는 실데이터화
- [ ] **오리지널 가이드 콘텐츠 엔진**: `/guides/*` 구조 + 초기 가이드 10편 이상
      (직접예약 방법, 다낭/하노이/호이안 도시 가이드, 베스트 시즌, 환불·취소 체크리스트 등)
- [ ] 각 페이지 충분한 유니크 텍스트(스캐폴드가 아닌 실제 문장)
- [ ] picsum → 실사진(Unsplash) 확대

### Phase 1 — 법적 페이지 + 기술 SEO 🤖
- [ ] `/privacy` (쿠키·제3자 광고·Google·opt-out 의무 문구), `/terms`, `/contact`, About 보강
- [ ] `robots.txt`, `sitemap.xml`(라우트 자동 생성), 페이지별 title/description/OG 점검
- [ ] 푸터에 법적 페이지 링크, 일관된 내비

### Phase 2 — 도메인 & 호스팅 🧑(구매/결제) + 🤖(설정)
- [ ] 도메인 구매(예 `stayeasy.asia`) 🧑
- [ ] Vercel 또는 Cloudflare Pages에 커스텀 도메인 연결, HTTPS 🤖+🧑
- [ ] GA4(또는 Plausible) 설치 🤖 / 계정 🧑

### Phase 3 — 백엔드 실배포 (등록·문의 실제 동작) 🧑(호스팅) + 🤖(코드)
- [ ] Fastify+Prisma → Render/Fly 배포, Postgres 전환 🤖 / 결제·계정 🧑
- [ ] 등록/문의 폼 → API 연결, 이메일 발송 연동 🤖 / API 키 🧑
- [ ] 프론트 `VITE_API_URL` 연결(이미 어댑터 준비됨) 🤖

### Phase 4 — AdSense 신청 & 광고 게재 🧑(계정) + 🤖(통합)
- [ ] AdSense 계정 생성·사이트 등록 🧑
- [ ] 사이트 인증 스니펫 삽입 🤖
- [ ] 심사 통과 후 `ads.txt` 추가 🤖
- [ ] 광고 슬롯 컴포넌트(`<AdSlot/>`) — 콘텐츠와 명확히 구분, 반응형 🤖
- [ ] 정책 점검(광고 밀도·오인 유도 금지) 🤖

## 5. "지금 당장 코드로 시작할 수 있는 것" (Phase 0–1 = AdSense 차단요인 직접 해소)

1. **De-demo**: `(Sample)` 라벨·mock/demo 면책 제거, 콘텐츠 톤을 실서비스로
2. **법적 페이지 3종**: `/privacy`(AdSense 필수 문구 포함)·`/terms`·`/contact`
3. **오리지널 가이드 콘텐츠**: 직접예약·도시 가이드 다편 신규
4. **기술 SEO**: `robots.txt` + `sitemap.xml` + 메타 보강
5. **AdSlot 컴포넌트 스캐폴드**(승인 후 pub-id만 넣으면 동작)

> 도메인·AdSense 계정·백엔드 호스팅·실제 호텔 제휴 데이터는 **사용자 영역**.
> 그 외 코드/콘텐츠/구조는 전부 선제적으로 준비해 둘 수 있다.

## 5b. 환경변수 (Vercel/호스팅에서 설정)

코드는 아래 env가 없어도 깨끗하게 동작(기능 비활성). 값을 넣으면 자동 활성화된다.

| 변수 | 용도 | 예시 | 시점 |
|---|---|---|---|
| `SITE_URL` | sitemap/robots 절대 URL | `https://stayeasy.asia` | 빌드 |
| `VITE_ADSENSE_CLIENT` | AdSense 광고 + `ads.txt` 활성화 | `ca-pub-XXXXXXXXXXXX` | 승인 후 |
| `VITE_GA_ID` | Google Analytics 4 | `G-XXXXXXXXXX` | 언제든 |
| `VITE_CONTACT_EMAIL` | 문의 메일 주소 | `hello@stayeasy.asia` | 도메인 후 |
| `VITE_API_URL` | 백엔드 API 연결(선택) | `https://api.stayeasy.asia` | 백엔드 배포 후 |

- 광고는 `VITE_ADSENSE_CLIENT` 설정 전엔 렌더되지 않음(정책 안전). 설정 시 `<AdSlot>`가
  스크립트를 로드하고 `ads.txt`도 빌드시 올바른 라인으로 생성된다.
- 분석은 `VITE_GA_ID` 설정 시 SPA 라우트 변경마다 page_view 전송.
- 구조화 데이터(JSON-LD: Article/Hotel/FAQ)는 항상 출력 — SEO·품질 신호.

## 6. 리스크 & 솔직한 한계

- **가짜 호텔로는 절대 통과 못 함** — 핵심 콘텐츠를 "오리지널 가이드"로 전환하는 게
  유일하게 확실한 경로. 호텔 디렉터리는 보조(스폰서/실제 제휴 시 확장).
- **승인까지 수일~수주**, 1~2회 거절은 흔함 → 콘텐츠/법적 페이지를 탄탄히 한 뒤 신청.
- **도메인 없이는 시작 불가** — Phase 2의 도메인 구매가 사실상 첫 외부 의존성.
- 광고와 스폰서 콘텐츠의 **구분 표기**를 처음부터 지켜야 정책 위반을 피한다.
